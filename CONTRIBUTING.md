# Contributing to AI Revenue Recovery

First of all, thank you for considering contributing to AI Revenue Recovery! 🎉

Community contributions help improve the project for everyone. Whether you're fixing a typo, improving documentation, reporting bugs, proposing new ideas, or implementing features, your help is appreciated.

---

# Ways to Contribute

There are many ways to contribute:

* Report bugs
* Suggest new features
* Improve documentation
* Fix existing issues
* Add tests
* Improve performance
* Refactor code
* Review pull requests
* Help answer questions in discussions

No contribution is too small.

---

# Before You Start

Before starting work on a large feature:

1. Search existing Issues to see if someone is already working on it.
2. If no issue exists, open one describing your proposal.
3. Wait for feedback before investing significant development time.

This helps avoid duplicate work and ensures the feature aligns with the project's direction.

---

# Development Setup

1. Fork the repository.
2. Clone your fork.

```bash
git clone https://github.com/<your-username>/AI-Revenue-Recovery.git
cd AI-Revenue-Recovery
```

3. Create a new branch.

```bash
git checkout -b feature/my-new-feature
```

4. Install project dependencies according to the README.

5. Verify everything builds successfully before making changes.

---

# Branch Naming

Please use descriptive branch names.

Examples:

```
feature/add-openai-provider
feature/improve-dashboard
bugfix/fix-auth-error
docs/update-installation
refactor/config-cleanup
test/add-api-tests
```

---

# Coding Guidelines

Please keep contributions:

* Simple
* Readable
* Well documented
* Consistent with the existing code style
* Focused on one logical change

Avoid unrelated formatting-only commits.

---

# Commit Messages

Use clear commit messages.

Examples:

```
feat: add retry support for API calls

fix: handle missing environment variables

docs: improve installation guide

refactor: simplify prompt builder

test: add unit tests for parser
```

Small, focused commits are preferred over one large commit.

---

# Pull Request Guidelines

Before submitting a Pull Request:

* Ensure the project builds successfully.
* Run all available tests.
* Update documentation if necessary.
* Add tests for new functionality when appropriate.
* Keep the PR focused on a single topic.

When opening the PR, include:

* What changed
* Why it changed
* Screenshots (if UI changes)
* Related Issue (if applicable)

Example:

```
Closes #42
```

---

# Reporting Bugs

When reporting bugs, please include:

* Operating system
* Runtime/version information
* Steps to reproduce
* Expected behavior
* Actual behavior
* Error messages
* Screenshots or logs (if helpful)

The more information you provide, the easier it is to reproduce and fix the issue.

---

# Suggesting Features

Feature requests should explain:

* The problem you're trying to solve
* Your proposed solution
* Possible alternatives
* Additional context

Well-described feature requests are much easier to evaluate.

---

# Documentation Contributions

Documentation improvements are always welcome.

Examples include:

* Fixing typos
* Improving examples
* Clarifying explanations
* Adding tutorials
* Updating screenshots

---

# Testing

Whenever possible:

* Add tests for new functionality.
* Ensure existing tests continue to pass.
* Avoid reducing test coverage.

---

# Code Review

All submissions go through code review.

Reviewers may request:

* Code improvements
* Additional tests
* Documentation updates
* Refactoring for consistency

Please don't take review comments personally—they're intended to improve the project.

---

# Code of Conduct

Please be respectful and constructive.

We strive to maintain a welcoming, inclusive, and collaborative community.

Examples of expected behavior:

* Be respectful.
* Assume good intentions.
* Give constructive feedback.
* Welcome newcomers.
* Keep discussions professional.

Harassment, discrimination, or abusive behavior will not be tolerated.

---

# Questions

If you're unsure where to begin, feel free to:

* Open a Discussion
* Open an Issue
* Ask for guidance before starting work

We're happy to help contributors get started.
